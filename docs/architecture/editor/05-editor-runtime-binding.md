# Editor runtime binding

The authoritative client source is `ClientReplicaStore`. `ReplicaEditorReadModel` projects replicas, capability-derived runtime actions, diagnostics, and local preview values without changing replica revisions.

Dear ImGui submits only capability action IDs. `EditorCommandGateway` delegates those IDs to `RuntimeActionResolver`; the frontend never imports or constructs concrete runtime commands. Durable intents then cross `EditorCommandGateway` into `ClientEditorCommandTransport`. The transport adds actor identity and expected object/state revisions, encodes an allow-listed `CommandRequestMessage`, and sends it through Fabric C2S networking. The server `AuthoritativeCommandGateway` validates and commits through `RuntimeTransaction`; responses are correlated by command ID. Accepted changes become visible only after Snapshot or Delta replication updates the client store. Rejections produce editor diagnostics.

Implemented and client-integrated: typed parameter commit, Open/Close/Lock commands, Accepted/Rejected response routing, and replica-driven refresh. Revision conflicts automatically request an Object Snapshot and retain the local preview until resync completes. Remaining gaps are accepted-command History population, replica removal selection cleanup, and world picking integration.
## Replication performance debt

Accepted parameter edits currently broadcast a complete Object Snapshot. This is the convergence-first K2.3 implementation, not the intended high-frequency protocol. Before multi-selection, gizmo transforms, wall dragging, large component graphs or high-player-count editing, introduce a revision-checked `ParameterDeltaMessage` or a generic `ObjectPatchMessage`. The patch must preserve ordered authority, conflict detection, resync fallback and the rule that Preview is cleared only after the authoritative update is installed.
